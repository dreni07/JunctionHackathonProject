00 / Executive overview

A brain, not a chatbot.

Pyramid AI is a multi-agent decision system — not a single LLM prompt. Specialized agents each hold a narrow mandate, invoke their own toolset, publish structured decisions into a shared orchestration layer, and a fusion engine merges those decisions into one venue recommendation the user can trust. Every layer is observable, replayable, and continuously improved through reinforcement learning on real booking outcomes.

01 / LangChain orchestration layer

LangChain as the conductor.

Every agent is a LangChain Runnable registered in a central orchestrator. The orchestrator wires agents into a directed acyclic graph (DAG): each node is an agent invocation, each edge is a conditional hand-off based on tool output. LangChain's AgentExecutor manages the tool-calling loop per agent, while our custom LangGraph state machine coordinates cross-agent transitions.

02 / Agent state · Redis

Redis: the agent's working memory.

Agent conversations, tool results, and intermediate decisions cannot live in PHP memory alone — voice sessions run for minutes and span multiple agent hand-offs. We persist everything in Redis with namespaced keys for O(1) look-ups.

◆ What we store

  • session:{uuid} — full conversation transcript + active agent pointer
  • tools:{uuid}:{agent} — serialized tool call results (JSON)
  • decisions:{uuid} — ranked venue list, selected space, pricing snapshot
  • rl:feedback:{event_id} — outcome signals for the reinforcement loop

◆ Why Redis

  • Sub-millisecond reads — voice agents can't wait on disk I/O
  • TTL-based session expiry — stale sessions auto-clean after 30 min
  • Pub/Sub for real-time ops dashboard updates when agents complete
  • Atomic counters for rate-limiting tool calls per session
GET session:a3f8… → { agent: "EventIntakeAgent", step: 4, ranked: [...], pricing: {...} }
SETEX decisions:a3f8… 1800 {"selected_space_id":"…","confidence":87.3,"available":true}

06 / Confidence scoring algorithm

Every venue gets a score.

The matching engine scores each commercial space on a 0–100 confidence scale. Spaces scoring zero are excluded (TUMO education zone, over-capacity). The algorithm balances capacity fit (snug rooms score higher than cavernous ones) against functional type compatibility (a concert prefers Event Hall; a workshop prefers Tech Lab).

confidence = (capacityScore × 0.55 + typeScore × 0.45) × 100
capacityScore = attendees ÷ space.capacity (clamped 0–1) — a 80-person event in a 100-cap room scores 0.80; in a 500-cap room scores 0.16
typeScore = 1.0 if functional_type ∈ preferred[event_type], else 0.45 (workable but not ideal)
Hard filters: zone_class = TUMO → 0 · attendees > capacity → 0 · capacity ≤ 0 → 0

◆ Example: conference · 120 pax

  • Event Hall (300 cap, Event Hall type) → 78.0
  • Tech Lab (80 cap) → excluded (over capacity)
  • Lounge (40 cap) → excluded

◆ Example: workshop · 25 pax

  • Classroom/Workshop (30 cap) → 91.3
  • Tech Lab (40 cap) → 79.4
  • Event Hall (300 cap) → 49.6 (type mismatch penalty)

◆ RL adjustment

  • Accepted bookings boost typeScore weights for that event_type
  • Rejected suggestions decay capacityScore bias over time
  • Re-scored nightly from PostgreSQL outcome table
07 / Calendar availability engine

Ranked list meets live calendar.

Scoring alone doesn't book a room — the SchedulingAgent walks the ranked list against the live reservation calendar stored in PostgreSQL. For each space it checks blocking reservations and venue unavailability records (maintenance, private holds, out-of-service blocks).

◆ Conflict detection

  • Query reservations where space_id matches AND time window overlaps AND status = blocking
  • Query venue_unavailabilities for maintenance / hold / out-of-service blocks
  • First space with zero conflicts → selected
  • If top-ranked space is blocked, report top_unavailable with human-readable reason

◆ Walk-through example

  • Rank #1 Event Hall (87.3) → booked (existing reservation 14:00–18:00)
  • Rank #2 Tech Lab (72.1) → maintenance hold
  • Rank #3 Event Space (68.5) → free → selected

08 / Dynamic pricing engine

Price from evidence, not guesses.

The SuggestPricingTool invokes the pricing service, which learns from a growing pricing reference dataset in PostgreSQL — past events, accepted agent suggestions, and market comps ingested by the ResearcherAgent. Pricing is per m² per day, scaled to the selected venue's area and event duration.

Every accepted price is written back to the reference table — the dataset grows with every confirmed booking, tightening future suggestions.

11 / Reinforcement learning loop

The system learns from its own decisions.

Every agent suggestion that becomes a confirmed booking generates a positive reward signal. Rejected proposals, revised dates, and cancelled events generate negative or neutral signals. These feed a nightly batch job that adjusts scoring weights and enriches the pricing reference dataset.

Agent decision
venue + price + confidence logged to Redis + PostgreSQL
→
Outcome event
confirmed · revised · rejected · cancelled
→
Reward signal
+1.0 confirm · −0.5 revise · −1.0 reject
→
Weight update
capacity/type weights + pricing comps adjusted

◆ Training data sources

  • Confirmed event requests (venue, price, dates, headcount)
  • PricingReference table (grows with every accepted suggestion)
  • ResearcherAgent market comps (web-scraped benchmarks)
  • Operational feedback (TaskPlanningAgent completion rates)

◆ What gets updated

  • Preferred functional_type maps per event_type
  • Capacity score bias (snug-fit preference strength)
  • PricingReference averages and comparable filters
  • Agent system prompt examples (few-shot injection)

◆ Guardrails

  • Only confirmed-closed events become training data
  • Minimum sample_size = 3 before weight shift applies
  • Human ops can override weights via config (no deploy)
  • LangSmith traces retained for audit replay

15 / Observability & audit

Every decision is traceable.

LangSmith captures the full agent execution graph: which tools were called, with what arguments, what they returned, and how long each step took. Combined with Laravel Pulse and PostgreSQL audit logs, ops can replay any booking decision end-to-end.

◆ LangSmith trace

input: { event_type: "conference", attendees: 120 }
output: [ 5 venues, top confidence 87.3 ]
latency: 342ms · tokens: 1,847

◆ Audit guarantees

  • Every tool call persisted with timestamp + session_id
  • Agent decisions immutable once submitted (append-only log)
  • RL feedback gated on confirmed-closed events only
  • Ops can export full decision chain for any event request
16 / Security model

Agents are sandboxed.

◆ Tool isolation

  • Each agent has a fixed tool allowlist enforced by LangChain executor
  • DbQueryTool is read-only — no agent can mutate data directly
  • api_submit is the only write path, gated on user confirmation

◆ Session security

  • Redis sessions bound to authenticated user_id
  • Rate limiting on tool calls per session (Cloudflare + app layer)
  • Organizer routes isolated from ops routes via middleware

◆ Edge protection

  • Cloudflare DDoS mitigation on all endpoints
  • WAF rules block SQL injection, XSS at the edge
  • TLS 1.3 everywhere · HSTS enforced
